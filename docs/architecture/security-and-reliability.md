# 安全与可靠性

## 1. 安全目标

平台涉及三类高敏数据：

1. MT5 账户凭证
2. 交易指令与风控配置
3. 用户身份、Token 与审计记录

设计原则是：凭证最小暴露、命令可追责、默认拒绝、链路可回放。

## 2. MT5 凭证保护

### 2.1 分级存储

建议将 MT5 凭证拆成不同级别：

1. 只读凭证
2. 交易凭证
3. 券商服务器连接信息
4. 平台内部派生令牌

### 2.2 存储原则

1. MySQL 中不保存明文密码。
2. 使用信封加密：字段级 AES-256 + 外部 KMS/HSM 主密钥。
3. 应用实例只持有短期解密权限，不持久化明文。
4. 所有凭证读取都必须写审计日志。

### 2.3 使用原则

1. 凭证只允许在 Account/Config Service 和 MT5 Bridge 的授权路径中解密。
2. 业务日志、异常日志、埋点日志中禁止输出密码、Token、完整账号。
3. 对外返回时统一脱敏，例：仅保留账号尾号和服务器别名。

## 3. 认证与授权

### 3.1 用户侧

1. Web/H5 使用 JWT Access Token + Refresh Token。
2. 高风险操作要求二次认证，至少支持 MFA 或签名确认。
3. 管理后台与普通用户权限隔离。

### 3.2 服务间

1. 内部 RPC / MQ 消费者身份使用服务证书或服务 Token。
2. 网关到后端禁止匿名透传。
3. Agent Service 不能绕过权限系统，所有动作仍走内部授权层。

## 4. WebSocket 安全

MT5 与平台通信使用 WebSocket 时，建议执行以下约束：

1. 生产环境优先 `wss://`。
2. 握手时携带短期 Bearer Token 或签名头。
3. 建立连接后，服务端将连接绑定到唯一 `login + server + client_instance_id`。
4. 对相同账号的重复连接设置接管或拒绝策略，避免双写。

## 5. 可靠性要求

## 5.1 事件语义

建议明确采用“至少一次投递 + 服务端幂等”的语义：

1. MT5 EA 断线重连时允许重复发送。
2. 服务端不得假设消息天然只到一次。
3. Copy Engine 和执行链路都要使用 `event_id` 或命令 ID 去重。

## 5.2 失败隔离

1. WebSocket 接入故障不应拖垮跟单引擎。
2. Copy Engine 计算失败不应阻塞其他主账号分区。
3. 某券商执行异常应局部隔离到对应 Bridge Worker。

## 5.3 重试规则

1. 网络瞬时失败可以重试。
2. 风控拒绝、余额不足、参数非法不重试。
3. MQ 消费失败进入延迟重试或死信队列，禁止无限自旋。

## 6. 一致性策略

交易系统不追求跨服务强一致，优先采用“最终一致 + 核心链路幂等”：

1. 配置修改以 MySQL 为准，异步刷新 Redis。
2. Redis 失效时可回源 MySQL，但不能成为常态。
3. 订单执行事实以 MT5 回执为准，监控快照以事件重放修正。

## 7. 审计要求

以下操作必须记录审计日志：

1. 绑定或修改 MT5 账户
2. 修改跟单规则
3. 启停跟单
4. 强制平仓
5. Agent 触发的自动动作
6. 凭证读取或解密

审计记录至少包含：

1. 操作者
2. 目标资源
3. 操作类型
4. 请求参数摘要
5. 结果
6. 时间
7. 关联追踪 ID

## 8. 可用性与恢复

建议第一阶段目标：

1. 核心信号接入可用性 `99.9%`
2. 监控查询可用性 `99.5%`
3. 关键配置备份周期 `<= 1h`
4. 恢复演练至少覆盖 Redis 缓存丢失、MQ 堆积、Bridge 重启、单实例故障

## 9. 第一阶段必须先做的保护项

即使第一版功能极简，下列能力也不要省略：

1. 凭证加密
2. 事件幂等
3. 跟单防环
4. 命令审计
5. WebSocket 身份校验
6. 服务级 trace id
