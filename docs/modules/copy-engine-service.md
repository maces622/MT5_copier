# 实时跟单引擎

## 当前实现状态

当前仓库已落地第一版最小闭环，覆盖：

1. 监听已接收的 MT5 `DEAL` 事件
2. 根据主账号 `server + login` 映射平台内部 master account
3. 读取 active 跟单关系
4. 生成本地 `execution_commands` 记录
5. 对固定手数、倍率手数、跟随原手数做基础换算
6. 对品种白黑名单、最大手数、参数缺失做拒绝记录

当前仓库尚未覆盖：

1. MQ 投递
2. 保证金检查
3. 滑点控制
4. 真实 MT5 下单
5. 执行回执回写

## 1. 职责

Copy Engine 是系统的核心决策层，负责回答三个问题：

1. 谁需要跟这笔单
2. 每个跟单账户该跟多少
3. 哪些账户应该被风控拦截

## 2. 输入

主要输入 Topic：

1. `signal.deal.v1`
2. `signal.order.v1`
3. `signal.equity.v1`
4. `copy.relation.changed.v1`
5. `risk.rule.changed.v1`

高频读依赖：

1. Redis 路由表
2. Redis 净值/保证金快照
3. Redis 去重键

## 3. 输出

1. `execution.command.v1`
2. `monitor.alert.v1`
3. `audit.action.v1`

## 4. 核心处理流程

以主账号成交事件为例：

1. 消费 `DEAL` 事件
2. 基于 `event_id` 去重
3. 从 Redis 读取主账号路由表
4. 并行计算每个跟单账户的目标手数
5. 校验风控约束
6. 生成执行命令
7. 按目标账户分组后写入执行 Topic

## 5. 手数计算策略

第一阶段建议先支持以下模式：

1. 固定手数
2. 按余额比例
3. 按净值比例
4. 跟随主账号原始手数

输出手数前还要统一经过：

1. 最小手数校正
2. 步长对齐
3. 最大手数截断
4. 账户可用保证金检查

## 6. 风控拦截点

建议按顺序执行：

1. 账户状态是否允许交易
2. 品种是否在白名单/黑名单中
3. 单笔手数是否超过上限
4. 当前回撤和日亏是否超限
5. 保证金是否足够
6. 是否触发价格偏差阈值

## 7. 并发与削峰设计

高并发场景下不要逐个账户同步 RPC，下列原则必须成立：

1. 主账号事件先入 MQ
2. 同一主账号单分区顺序消费
3. 单次事件内部对跟单账户并发计算
4. 执行命令批量投递，不阻塞信号主线程

可以接受“部分跟单账户稍后几十毫秒执行”，不能接受“单个主账号洪峰拖垮全局”。

## 8. 幂等模型

建议使用两级键：

1. 事件级：`master_event_id`
2. 命令级：`master_event_id + follower_account_id + action_hash`

这样可避免：

1. 上游重复发信号
2. 引擎重平衡或消费重试造成重复下单

## 9. 第一阶段边界

先做最小闭环，不要一开始就处理所有复杂交易场景。第一阶段只做：

1. 市价单开仓/平仓
2. 固定手数和比例手数
3. 基础保证金检查
4. 简单止损止盈跟随

挂单、反向跟单、分批成交、复杂价格保护留到后续迭代。
