# 迭代工作流

## 1. 工作原则

后续所有功能都采用固定循环推进：

1. 设计
2. 实现
3. 验证
4. 调整设计

任何一轮迭代都必须明确边界，禁止在没有设计收敛的情况下同时铺开多个高风险模块。

## 2. 单轮迭代模板

### 2.1 设计

输出物：

1. 目标问题
2. 范围与非范围
3. 接口或消息契约
4. 数据模型
5. 风险点
6. 验收标准

设计完成后，需要先把相关文档更新到 `docs/`。

### 2.2 实现

规则：

1. 代码只实现本轮设计范围
2. 不提前埋入大面积未验证抽象
3. 关键路径优先可观测性和错误处理

### 2.3 验证

至少覆盖三类验证：

1. 正常路径
2. 边界条件
3. 失败/恢复路径

对跟单系统来说，验证不只是单元测试，还包括：

1. WebSocket 联调
2. MQ 顺序与重复消费验证
3. Redis 缓存失效回退验证
4. 风控拒绝验证

### 2.4 调整设计

验证后必须回写文档，更新：

1. 假设是否成立
2. 接口是否需要变更
3. 是否新增风险项
4. 下一轮要收敛的问题

## 3. Definition of Done

一轮任务完成，至少满足：

1. 文档已更新
2. 代码已实现
3. 验证结果已记录
4. 未解决问题已列出

如果只做了代码、没有补设计或验证记录，该轮不算完成。

## 5. 每轮交付时的固定问题

每次进入下一轮前，都先回答：

1. 这一轮的输入契约是什么
2. 这一轮的输出契约是什么
3. 失败了如何恢复
4. 如何验证没有重复下单
5. 如何确认文档仍和代码一致
